# 2.2 — Escrita no banco a cada requisição (IMPORTANTE)

## Problema
`validate()` é `@Transactional` e muta `lastAccessedAt`
(`service/SpringSessionLiteService.java:43-58`) → UPDATE por requisição autenticada. O caminho
de leitura virou caminho de escrita.

## Impacto
Gargalo de throughput sob carga; impede réplica de leitura; contenção de locks na tabela.

## Risco
Médio (performance/escalabilidade).

## Arquivos
- `service/SpringSessionLiteService.java`
- `config/SpringSessionLiteProperties.java` (novas flags)

## Solução / passos
1. Adicionar propriedade `update-last-accessed` (default `true`) e
   `last-accessed-throttle` (Duration, ex. `5m`).
2. Em `validate`: só atualizar `lastAccessedAt` se `agora - lastAccessedAt > throttle`
   (reduz writes drasticamente). Se `update-last-accessed=false`, leitura pura.
3. Quando não houver mutação, marcar a transação `readOnly = true` (ou separar método de
   leitura sem `@Transactional` de escrita).
4. Coordenar com 3.2 (sliding expiration) caso ambos sejam implementados — mesma lógica de
   "estender com throttle".

## Complexidade
Média

## Verificação
- Teste: N requisições dentro do throttle geram **1** UPDATE (mock/contagem no repo).
- Teste: `update-last-accessed=false` → zero UPDATE no caminho de leitura.
- `./mvnw test` verde.
