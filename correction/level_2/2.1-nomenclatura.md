# 2.1 — Nomenclatura inconsistente (IMPORTANTE)

## Problema
Três marcas convivem: prefixo `SpringSessionLite*` (maioria) vs `SpringLite*` (entidade/repo),
e branding residual `m4all`: cookie `M4SID`, tabela `spring_lite_sessions` (código) vs
`m4_sessions` (DDL/docs).

## Impacto
API incoerente, autocomplete ruidoso, doc divergente, confusão de manutenção.

## Risco
Médio (manutenção/DX).

## Arquivos (rename amplo)
- `domain/SpringLiteSession.java`, `domain/SpringLiteSessionRepository.java` → prefixo único
- Todas as classes `SpringSessionLite*` (revisar coerência)
- `config/SpringSessionLiteProperties.java` (default `cookie-name`)
- `domain/SpringLiteSession.java` (`@Table name`, índices)
- `resources/db/*-schema.sql`, `docs/*`
- imports em `META-INF/spring/...AutoConfiguration.imports`

> **Onda 0 — base de tudo.** Rodar ANTES de 1.1 (DDL), 1.2 (docs) e de itens que tocam nomes
> públicos. É a fundação de coesão.

## Solução / passos
1. Decidir prefixo único definitivo (sugestão: manter `SpringSessionLite*` em TODAS as
   classes, inclusive entidade/repo → `SpringSessionLiteSession`/`...Repository`, ou encurtar
   tudo para `SessionLite*`). **Decisão precisa ser tomada com o autor.**
2. Renomear classes/arquivos de forma consistente (IDE refactor ou rename manual + ajuste de
   imports).
3. Cookie default neutro (ex.: `SLSID`) — **breaking** para apps já usando `M4SID`; manter
   configurável (já é).
4. Tabela única e coerente (ex.: `spring_session_lite_sessions` ou `session_lite`). Alinhar
   `@Table`, índices, DDL.
5. Atualizar `AutoConfiguration.imports` se o FQN da classe de auto-config mudar.

## Complexidade
Baixa (rename de classes) / Média (renomear tabela e cookie após qualquer adoção).

## Verificação
- `grep -rn "m4\|M4SID\|SpringLite[^S]"` → vazio (sem prefixos mistos/branding antigo).
- `./mvnw test` verde (ajustar testes que referenciam `M4SID`).
