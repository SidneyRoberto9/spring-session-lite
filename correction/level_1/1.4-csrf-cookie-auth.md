# 1.4 — CSRF desabilitado com auth por cookie (CRÍTICO)

## Problema
O chain padrão faz `csrf(disable)` (`autoconfigure/SpringSessionLiteAutoConfiguration.java:112`)
e a doc instrui o mesmo. Diferente de JWT em header, o cookie `M4SID` é enviado
automaticamente pelo browser → requisições que mudam estado ficam expostas a CSRF.
`SameSite=Lax` mitiga parcialmente, mas a lib permite `SameSite=None`, reabrindo o vetor.

## Impacto
Apps que usem o default com `SameSite=None` ficam vulneráveis a CSRF.

## Risco
Alto (segurança).

## Arquivos
- `autoconfigure/SpringSessionLiteAutoConfiguration.java`
- `config/SpringSessionLiteProperties.java` (nova flag)
- `docs/01-instalacao-e-uso.md` (trade-off documentado)

## Solução / passos
1. Adicionar propriedade `csrf-enabled` (default conforme decisão; sugestão: manter compat
   com `false`, mas documentar fortemente).
2. **Fail-fast / aviso no startup:** se `cookie-same-site=None` e CSRF desabilitado, logar
   `WARN` explícito (ou recusar) — combinação insegura.
3. Oferecer caminho seguro: integração opcional com `CookieCsrfTokenRepository.withHttpOnlyFalse()`
   quando `csrf-enabled=true`, OU defesa simples por header custom (`X-Requested-With`) exigido
   em métodos mutantes.
4. Documentar: recomendar `SameSite=Lax`/`Strict`; explicar quando habilitar CSRF.

## Complexidade
Média

## Verificação
- Teste: com `csrf-enabled=true`, requisição mutante sem token CSRF → 403.
- Teste: `SameSite=None` + CSRF off → log de WARN presente (ou boot recusado).
- `./mvnw test` verde.
