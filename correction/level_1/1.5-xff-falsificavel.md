# 1.5 — X-Forwarded-For usa IP falsificável (CRÍTICO)

## Problema
`SpringSessionLiteIpResolver.resolve` usa `xff.split(",")[0]` — o IP mais à esquerda, que é
**declarado pelo cliente** (`service/SpringSessionLiteIpResolver.java:18`). Com um proxy real,
o atacante envia seu próprio `X-Forwarded-For`; o proxy só acrescenta o IP dele à direita, e o
leftmost continua sendo o valor do atacante.

## Impacto
Com `trust-forwarded-for=true`, o IP-binding passa a ser controlado pelo atacante → anula a
proteção que deveria oferecer.

## Risco
Alto (segurança), condicionado a `trust-forwarded-for=true`.

## Arquivos
- `service/SpringSessionLiteIpResolver.java`
- `config/SpringSessionLiteProperties.java` (nova propriedade)

## Solução / passos
1. Adicionar propriedade `trusted-proxy-count` (int, default 1) OU adotar estratégia baseada
   em proxies confiáveis.
2. Em `resolve`, ao confiar no XFF, pegar o IP na posição **`n - trustedProxyCount`** a partir
   da direita (o primeiro IP **não** adicionado por proxy confiável) — nunca o leftmost cego.
3. Alternativa preferida: delegar ao mecanismo nativo do Spring
   (`server.forward-headers-strategy=FRAMEWORK`/`ForwardedHeaderFilter`) e usar
   `request.getRemoteAddr()` já normalizado — documentar essa via.
4. Documentar que leftmost é sempre não-confiável e exigir proxy que **sobrescreva** XFF.

## Complexidade
Média

## Verificação
- Teste do `IpResolver`: `X-Forwarded-For: 9.9.9.9, 10.0.0.1` com 1 proxy confiável → retorna
  `9.9.9.9` (cliente real), não o valor injetável adicional.
- Teste: XFF spoofado com proxy confiável não permite forjar o IP do binding.
- `./mvnw test` verde.
