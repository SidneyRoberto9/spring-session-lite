# 3.4 — `SpringSessionLiteUser` como record + principal extensível (OPCIONAL)

## Problema
Classe Lombok `@Getter/@AllArgsConstructor`; expõe o principal como corpo de resposta,
acoplando o contrato de API ao principal.

## Impacto
Boilerplate, mutabilidade desnecessária, acoplamento do contrato.

## Risco
Baixo.

## Arquivos
- `security/SpringSessionLiteUser.java`
- testes/docs que constroem/leem o user

## Solução / passos
1. Converter para `record SpringSessionLiteUser(String userId, String email, String sessionId, ...)`.
   - **Coordenar com 2.4 (roles)** — se ambos, adicionar `roles` ao record de uma vez.
2. Garantir compat de serialização Jackson (record é suportado nativamente em Boot 3).
3. (Opcional) claims extras via `Map<String,Object>` ou tipo genérico, para extensibilidade.
4. Ajustar `getUserId()`/`getEmail()` → acessores de record (`userId()`...) ou manter compat com
   métodos — avaliar impacto nas docs.

## Complexidade
Baixa

## Verificação
- `./mvnw test` verde (deserialização de `SpringSessionLiteUser` no `TestRestTemplate`).
